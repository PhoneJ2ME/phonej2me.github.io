Participation Guidelines

The Mobile & Embedded open source community welcomes and encourages your participation and contributions. The participation guidelines help you to understand how ideas and code are discussed, developed, contributed, and maintained within the community.

Note: The guidelines are a work in progress and will be revised and improved as the community and the core projects evolve.

The Governance Model and the Development and Contribution Process

At a high level, participation in the community and code development is defined by the following guidelines:

  • Community-wide governance model

    The governance model defines the purpose of the community, the principles, the rules of community participation, and the roles and responsibilities of those who participate. It outlines the framework for conducting community business.

  • Project-specific development and contribution process (see the Getting Started guide for the project in which you are interested)

    The development and contribution process defines the procedures, rules, guidelines and details of how the members of a project work together to create and improve the projects shared assets such as code and content. It relies on the framework of the governance model for community-wide guidelines.

Together, the governance model and the development and contribution process form the basis of how the project operates, creates, and improves code and content and achieves the goals of the project.

Defining the Process for Community Core Projects

The Mobile & Embedded open source community has a set of core projects: phoneME and cqME.

These guidelines describe the development and contribution process used for these core projects. Other projects can adopt all or parts of this process or define their own process depending on their needs. One of the goals of these guidelines is to provide a working example for other projects in the community. For information about other projects, see the documentation for that individual project.

The core projects are complex and have code bases with a long history and a large number of engineers working on them. As such, they require a fairly detailed and formalized development process to facilitate effective development. These guidelines describe the relevant definitions, elements, and steps to enable community members new to these projects to familiarize themselves with how the code is maintained and improved.

 

Goals of these Guidelines

These guidelines have the following goals:

  • Facilitate a maximum degree of openness and transparency in the development and contribution process of core projects.

  • Provide a common set of terms, processes, rules, guidelines, and templates for all members of the core projects.

  • Ensure technical excellence and the highest quality of contributions, code, and resulting releases.

  • Strike a balance between the need for formalized development and process overhead to facilitate efficient cooperation between all project participants.

  • Provide generally agreed upon best practices for all optional process elements.

  • Provide a working example for other projects in the community.

Audience

All community members who are interested in understanding the development and contribution process used by the core projects of the Mobile & Embedded open source community.

Prerequisites

These guides assume that you are familiar with the documents and information listed in the Reference section. It is also beneficial to have experience with software development in general as well as source code control systems and tools, particularly Subversion and Code Striker.

Definitions

Module
Logical unit of related functionality within a project. A list of modules and module owners can be found here <link to module owners>
 
Patch
Issue Tracker terminology for a contribution.
 
Change Request (CR)
Single defect or request for enhancement (RFE) in Issue Tracker.
 
Change Package
Package of files that contains all of the information that is related to a contribution and can be applied directly to a repository through a source code management mechanism.
 
RFE
Request for enhancement.
